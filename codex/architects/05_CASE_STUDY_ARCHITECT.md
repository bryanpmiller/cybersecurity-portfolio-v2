# 05_CASE_STUDY_ARCHITECT.md - Case Study Experience

## Purpose

Use this architect when a task touches project cards, project detail pages, evidence blocks, GitHub proof links, technical visuals, screenshots, or project storytelling.

The goal is to make projects feel like operational case studies, not simple GitHub cards.

---

## Case Study Principle

Every project should quickly communicate:

```text
Problem -> Approach -> Evidence -> Outcome -> What it demonstrates
```

Not every card needs all five in full detail, but the structure should be clear somewhere in the project experience.

---

## Project Card Rules

Cards should help recruiters and technical reviewers decide what to inspect.

Cards should expose:

- Problem
- Outcome
- Concepts
- Tools
- Link to the full case study
- Link to GitHub proof when available

Cards should not become:

- Too dense
- Overloaded with badges
- Repetitive dashboards
- Long descriptions hidden in small text
- A wall of identical blocks

---

## Project Detail Page Rules

Detail pages should feel like evidence briefings.

Prioritize:

- Clear title and role
- Tools and environment
- Problem statement
- Approach/process
- Evidence chain
- Outcome
- Lessons or maturity signal
- GitHub/supporting proof links

Avoid:

- Unsupported claims
- Rewriting history to sound bigger than it was
- Hiding the actual work below too much visual polish
- Overusing screenshots without context
- Dumping raw technical strings without wrapping

---

## Evidence Rules

Evidence should be specific and supported.

Good evidence examples:

- Vulnerability reduction metrics
- STIG remediation artifacts
- KQL query logic
- Defender/Sentinel investigation pivots
- PowerShell remediation scripts
- GitHub folders
- Scan/export proof
- Timeline or workflow diagrams

Avoid:

- Generic "security dashboard" visuals
- Fake terminal output
- Invented detections
- Decorative charts without source
- Repeating the same tool list on every surface

---

## Screenshot and Visual Proof Rules

Screenshots and visuals should be presented like product/evidence surfaces.

Use:

- Framed visuals
- Captions or short context
- Clear connection to the case study
- Responsive-safe containers
- Selective placement

Avoid:

- Image dumps
- Repeated visuals
- Decorative screenshots with no explanation
- Visuals that push important text below the fold unnecessarily

---

## Review Questions

Before accepting a case-study change, ask:

- Does this make the project easier to understand?
- Does it strengthen credibility?
- Does it preserve supported claims?
- Does it reduce or increase clutter?
- Does it help a recruiter or technical reviewer choose to click deeper?
- Does it create duplicate proof nearby?

If the change looks nice but does not improve proof or clarity, reconsider it.
